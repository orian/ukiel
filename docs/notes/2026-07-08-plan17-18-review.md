# Post-landing review of plans 17/18 (2026-07-08)

High-level review of the day's commits (`db151db..04bf1ed`: fanout ladder,
size-targeted placement, cold-partition finalization, ingest backpressure,
assignment seam). No correctness bugs found — the exactly-once reasoning
around deferral, the shutdown backpressure bypass, and the finalization
TOCTOU recheck-then-conflict handling all check out. Five concerns, each
tracked as its own issue (0005–0009). Actionable ones (0005 interim, 0006,
0007, 0009) batch into roadmap row 28; 0008 is accepted with a defined
revisit trigger.

## 1. Whole-partition merges are in-memory → issue 0005, row 28 (interim)

`merge_parts` concatenates all inputs into one `RecordBatch` and sorts it in
RAM. Already a declared non-goal (lsm-hierarchy note: "streaming merge is
deferred"), but finalization raises the stakes: it feeds the *entire live
partition* (all levels) into one merge, and size-targeted placement
explicitly anticipates multi-hundred-MB outputs. A large cold partition is
now the compactor's peak-memory event — and because the sweep is stateless,
an OOM there becomes a crash loop (same merge replanned every restart).

Interim guardrail (row 28): skip merges whose input `sum(size_bytes)`
exceeds a configured cap; warn + metric. A too-big partition stays
unfinalized (correct, just not terminal) instead of crash-looping. Real fix
(streaming k-way merge over already-sorted inputs) stays deferred — details
in issue 0005.

## 2. Backpressure probe: one catalog query per buffered day per tick → issue 0006, row 28

`flush()` calls `live_l0_parts` once per buffered day, every flush tick.
Backfills are exactly when many days are buffered (that is what
`warn_partitions_per_flush` detects), so the probe is most expensive
precisely when the system is already stressed.

Fix: one grouped probe — `live_l0_counts(hypertable_id, partitions: &[Value])
-> Vec<(Value, i64)>` (single round-trip, `GROUP BY partition_values`);
`flush_decision` keeps taking the max. Must land before plan 8 ports the
backpressure code into `PipelineIngest`.

## 3. `partition_l0_quiet_since` scans all-history L0 → issue 0007, row 28

The quiet probe takes `max(created_at)` over every L0 part ever committed to
the partition — tombstoned included, which is deliberate (it measures
*arrivals*, not liveness), but no index serves
`(hypertable_id, partition_values, level)`: `parts_live_idx` is keyed on
packing-key range and partial on live rows only. Each finalize candidate
costs a per-hypertable scan, every sweep, growing with all-time history.

Fix: migration `0007` composite index
`parts (hypertable_id, partition_values, level)` — also serves
`live_l0_parts`, `live_partition_parts`, and `finalize_candidates`'
`GROUP BY`.

## 4. Flat 2× slowdown band → issue 0008 (accepted, revisit trigger defined)

The every-other-tick defer is a flat halving across the whole
`[l0_slowdown_parts, l0_stop_parts)` band. A proportional scheme would
degrade more gracefully when compaction falls far behind, but the flat rule
is easier to reason about and the memory valve bounds the damage. Accepted
as-is. Revisit trigger: metrics P2's backlog gauges (row 21) showing
sustained hover at `l0_stop_parts` — that is the signal the band is too
blunt, and P2 provides the data to tune a replacement.

## 5. Guardrail default triplication → issue 0009, row 28

The ingest guardrail defaults (30/200/64) live in three places: the
`IngestConfig` serde `default =` fns, `IngestSection::Default` in ukield,
and the ingest test `cfg()`. (The compactor's library-vs-ukield split — 2/2
test-compat vs 4/10 production — is intentional and commented; not this.)
Fix: ukield's `IngestSection::Default` reuses the `ukiel-ingest` default
fns (export them or a consts module) so the production defaults have one
source of truth; the test fixture can keep explicit values (its comment
says the bands should read clearly).
