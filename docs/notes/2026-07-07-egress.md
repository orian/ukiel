# Egress: parquet→kafka pipelines (sketch, roadmap row 24)

Status: sketch. The delivery ladder is already designed in
`docs/notes/2026-07-06-pipelines.md` (§Delivery semantics); this note
resolves the worker shape so the plan (written after plan 8) is mechanical.

## Worker shape

`EgressWorker` = the MV worker's skeleton with a Kafka producer target:
change-feed cursor over the source hypertable, **`kind == "add"` commits
only** (same double-count rule as MVs), batch = read parts → pipeline SQL →
produce one message per result row to the target `stream_table`'s topic.

- **v1 delivery = `at_least_once`**: produce the batch, await all acks,
  then `set_cursor`. Crash between acks and cursor = duplicates on replay,
  never loss. Every message carries headers `(commit_id, part_id,
  row_index)` so consumers *can* dedup ("effectively once").
- **Encoding v1**: JSON object per message from the SELECT's output schema
  (symmetric with kafka-source decode). Message key = the packing-key
  column's value if present in the output (gives consumers per-tenant
  partition affinity), else null.
- **`at_most_once`**: cursor first, then produce — config-selectable, no
  extra machinery.
- **`exactly_once`**: deferred until a consumer needs it; the pipelines
  note specs it (transactional producer + control-topic cursor,
  `read_committed` bootstrap). The catalog cursor becomes a mirror.

## The GC fence obligation (from the pipelines note, load-bearing)

The egress cursor is a `worker_cursors` row, so an egress pipeline
automatically fences the reaper: a stalled publisher's unread parts cannot
be reaped. In `exactly_once` mode the Postgres mirror may lag the
Kafka-committed truth — safe direction (fences more conservatively).
Consequence worth stating in the plan: **a dead egress pipeline blocks GC
for its hypertable** — the feed-lag alarm (monitoring) is the operational
counterweight, and row 23's horizon-margin gauge makes it visible.

## Error handling & flow control

`error_policy` stall/skip as everywhere. Producer-queue-full or broker
outage = stall (retry the batch; cursor pinned). Batch size bounded by the
existing `changes_since` limit; no new backpressure machinery needed —
egress is pull-based.
