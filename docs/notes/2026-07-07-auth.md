# Auth on the query endpoint (sketch, roadmap row 25 / issue 0001)

Status: sketch + decision framework. **Gated on a product decision, not on
other plans** — the open question is who issues credentials, which depends
on whether a SaaS control plane exists. Until this lands, the standing rule
holds: `namespace_id` is caller-supplied; do not expose the endpoint to
untrusted callers.

## Requirements

1M namespaces; stateless query nodes (per-request validation must not need
a catalog round trip on the hot path, or must cache hard); revocable;
principal → namespace binding is the *only* isolation input (the plan-time
filter does the rest — auth's job is exactly "which namespace is this?");
operator/system access distinct from tenant access; quotas want to ride the
same principal object later.

## Options

- **(a) Catalog API keys** — `namespace_keys(namespace_id, key_hash,
  created_at, revoked_at)`; bearer key → hash lookup → namespace, with an
  in-process LRU+TTL cache. Pros: self-contained, works for self-hosted and
  dev, revocation is a row update. Cons: ukiel becomes a credential store;
  cache TTL vs revocation latency trade-off; key issuance UX is on us.
- **(b) JWT from the SaaS control plane** — bearer token with an `ns`
  claim (+ `exp`, maybe `quota` claims), verified against JWKS fetched from
  config; zero catalog lookups, revocation via short TTL + issuer-side
  refusal. Pros: right shape at 1M tenants, keeps credentials out of
  ukiel, quotas/claims extensible. Cons: requires an issuer to exist;
  clock-skew and key-rotation operational surface.
- **(c) mTLS / network trust** — for *internal* callers only (system
  pipelines, operator tooling); never sufficient for tenant traffic.

## Recommendation

**(b) as the SaaS deployment's mechanism, (a) as the self-hosted/dev
fallback, both normalizing to one type** before session build:

```
AuthenticatedPrincipal { namespace_id: NamespaceId, /* later: quotas, audit id */ }
```

`run_query_inner` takes the principal instead of `req.namespace_id`
(**breaking change**: the field leaves the request body; the demo curl
gains an `Authorization: Bearer …` header). Session build and the plan-time
filter are untouched — auth is a thin adapter in `server.rs` plus config
(`[auth] mode = "jwt" | "api-key" | "insecure"`, with `insecure` printing a
loud warning and existing only for `make play`).

Audit: one structured log line per query with principal + namespace +
query class (never SQL text at info level).

## The decision needed before a plan

Does a control plane that can issue tenant JWTs exist (or will one)? If
yes → (b) primary. If ukiel must stand alone → (a) primary and the plan
adds key-management endpoints (create/revoke), which is meaningful extra
scope. Everything else above is settled either way.
