# NNNN — Concise issue title

- **Severity:** Critical | High | Medium | Low (brief rationale)
- **Status:** Open
- **Components:** `path/or/component`, `another/component`
- **Found by:** review, incident, benchmark, or test; YYYY-MM-DD

## Summary

Describe the observed behavior and the invariant, requirement, or expectation it
violates. Prefer concrete evidence: code paths, measured values, reproduction
conditions, or links to plans/specifications.

## Impact

Explain who or what is affected, how severe the failure is, and what remains
safe. Distinguish correctness, availability, performance, operability, and
documentation impact. State important scope limits so the issue is not read more
broadly than the evidence supports.

## Proposed resolution

Describe the smallest sound direction, including important constraints. Avoid
turning an issue into an implementation plan unless sequencing or alternatives
are essential to understanding the fix.

## Acceptance criteria

- Add observable, testable conditions that demonstrate the issue is resolved.
- Include regression coverage for the original failure mode.
- Name any performance, migration, compatibility, or documentation checks.

## Notes

- Record rejected shortcuts, follow-up work, and relationships to roadmap rows.
- When resolved, update **Status** with the plan/commit and summarize what
  actually shipped; keep the original problem statement for history.
