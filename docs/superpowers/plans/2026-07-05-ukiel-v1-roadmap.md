# Ukiel V1 Roadmap

Spec: `docs/superpowers/specs/2026-07-05-ukiel-design.md`. V1 is split into sequential plans, executed in order (rows 1–7, 9, 10, 11, 12, 17, 18, 19, 20, 21, 28, and 30 executed so far). Each plan produces working, tested software. Detailed plans are written just-in-time (after the previous plan merges) so interfaces referenced are real, not guessed.

**Recommended execution order for the remaining plans: 27 → 14 → 29 → 13 → 15 → 16 → 8, then 22 → 23 → 24** (25/26 gated on design decisions). Rationale: 28 (executed 2026-07-08) hardened 17/18 (grouped probe, partition index, merge cap, default dedup) ahead of 8's backpressure port; 21 first (pulled forward 2026-07-08 — its gates, 20 and ideally 17/18, are all executed): the regimes 17/18 introduced (ladder backlog, backpressure hover, unfinalized partitions, row-28's capped merges) are dark without P2's collector gauges — the design doc's "alarms standing in for hard limits" — issue 0008's revisit trigger depends on them, and the perf plans that follow churn exactly the paths those gauges watch (same empirically-check-what-follows rationale that put 20 before 17/18); 20 (metrics P1) shipped so its gauges empirically check everything after it; 11 built the perf harness/baseline before 17 changed the file population; 17 rewrote the compactor write path that 12/14/15/16/27 build on (landing it first avoided mutual adaptation and re-benchmarking); 18's backpressure now guards the fleet while the perf plans churn; 27 sits immediately before 14 because it births the shared sorting/writer-props module in `ukiel-core` that 14's key-aligned writer then extends (one module, not two) — it lands after 17 so it ports 17's rewritten compactor write path onto that module rather than forcing the already-written 17 to be re-authored, and its correctness fix is latent (no current config trips it, so no urgency pulls it earlier); 30 right after 21 — volume-scale baselines (10 GB ClickBench read suite, Bluesky write-path runs) must predate the write-path churn (27/14/29) and read-path churn (13/15/16) they will measure, the same baseline-before-churn rationale that put 11 before 17, and P2's gauges (21) instrument the runs; 29 sits right after 14 because streaming merge restructures the writer 14 builds and trusts the ordering 27 validates — and it is the PB-scale gate: the plan-28 cap keeps the process alive but leaves 100 GB+ partitions permanently multi-run, so at target scale (100–500 GB partitions/keys are common) the terminal invariant depends on 29, ahead of the read-side tail 13/15/16; 8 executes last because pipelines replace the ingest code 18 and 19 land in (its plan is written, with guardrail-continuity constraints). Rows 21+ are the post-v1 tail: written just-in-time once their gates land. Per-row notes below.
