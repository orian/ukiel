# Ukiel V1 Roadmap

Spec: `docs/superpowers/specs/2026-07-05-ukiel-design.md`. V1 is split into sequential plans, executed in order (rows 1–7, 9, 10, 11, 17, 18, 19, and 20 executed so far). Each plan produces working, tested software. Detailed plans are written just-in-time (after the previous plan merges) so interfaces referenced are real, not guessed.

**Recommended execution order for the remaining plans: 12 → 27 → 14 → 13 → 15 → 16 → 8, then 21 → 22 → 23 → 24** (25/26 gated on design decisions). Rationale: 20 (metrics P1) shipped so its gauges empirically check everything after it; 11 built the perf harness/baseline before 17 changed the file population; 17 rewrote the compactor write path that 12/14/15/16/27 build on (landing it first avoided mutual adaptation and re-benchmarking); 18's backpressure now guards the fleet while the perf plans churn; 27 sits immediately before 14 because it births the shared sorting/writer-props module in `ukiel-core` that 14's key-aligned writer then extends (one module, not two) — it lands after 17 so it ports 17's rewritten compactor write path onto that module rather than forcing the already-written 17 to be re-authored, and its correctness fix is latent (no current config trips it, so no urgency pulls it earlier); 8 executes last because pipelines replace the ingest code 18 and 19 land in (its plan is written, with guardrail-continuity constraints). Rows 21+ are the post-v1 tail: written just-in-time once their gates land. Per-row notes below.
